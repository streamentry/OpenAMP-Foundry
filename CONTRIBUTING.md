# Contributing

## Before opening a PR

1. Read `SAFETY.md`.
2. Read `RESPONSIBLE_USE.md`.
3. Run tests.
4. Avoid unsafe biological details.
5. Include limitations for new scoring or benchmark modules.

## PR requirements

A good PR includes:

- clear scope;
- tests;
- documentation update if behavior changes;
- safety impact note;
- license status for any data/model dependency.

## Safety impact note

Add this to PR descriptions:

```md
### Safety impact
- [ ] No biological misuse capability added
- [ ] No wet-lab protocol added
- [ ] No harmful optimization objective added
- [ ] Limitations documented
```
