# Adoption Metrics

## Purpose

This document defines how OpenAMP Foundry should measure adoption without confusing attention for trust.

The goal is not to maximize stars.

The goal is to become the default serious open infrastructure layer for trustworthy antimicrobial peptide experiment selection.

## Prime rule

**Measure external trust, artifact reuse, and evidence quality before popularity.**

## Weak metrics

Weak metrics can be useful but should not drive strategy:

- GitHub stars;
- social media mentions;
- demo views;
- number of generated candidates;
- number of models plugged in;
- number of claims made;
- apparent benchmark count without benchmark quality.

These are easy to inflate.

## Stronger metrics

Better signals:

- external contributors completing reviewable tasks;
- independent reruns of the demo or benchmarks;
- external use of evidence certificate schema;
- external use of dataset/model cards;
- external benchmark audits;
- reviewer rejections that improve the project;
- negative results safely preserved;
- release decisions recorded;
- calibration rejections recorded;
- downstream tools using OpenAMP artifacts;
- qualified experts reviewing candidate packages.

## Adoption levels

### Level 0 — Internal use

Maintainers can run the project.

Useful but not adoption.

### Level 1 — New contributor run

A new human or agent can install, run demo, inspect outputs, and identify a safe task.

### Level 2 — External contribution

A non-maintainer contributes tests, docs, schema, benchmark, or review improvements.

### Level 3 — External audit

An outsider reviews a benchmark, safety boundary, claim, or candidate package and leaves actionable critique.

### Level 4 — Artifact reuse

Another group uses OpenAMP artifact formats, even if it does not use OpenAMP scoring.

### Level 5 — External review workflow

A qualified outside reviewer can evaluate an OpenAMP packet without private context.

### Level 6 — Structured result feedback

Qualified result summaries can be ingested safely, interpreted conservatively, and used only through gates.

### Level 7 — Repeated learning loop

Multiple rounds show improved selection or decisive negative learning under pre-registered rules.

## Dashboard categories

A future adoption dashboard should track:

### Trust

- safety-reviewed releases;
- claim downgrades;
- decision records;
- release-status coverage;
- external review packets.

### Reproducibility

- successful first-run reports;
- deterministic output checks;
- schema-valid artifacts;
- benchmark reruns;
- CI health.

### Evidence

- benchmark cards;
- cheap-baseline comparisons;
- leakage checks;
- negative results;
- calibration gate outcomes.

### Ecosystem

- contributors;
- downstream artifact users;
- model/data cards;
- external adapters;
- issue completion by humans and agents.

### Safety

- blocked unsafe changes;
- staged/restricted release decisions;
- safety review issues;
- documentation safety audits;
- candidate/model release reviews.

## Adoption anti-patterns

Avoid:

- chasing stars with hype language;
- publishing candidate lists for attention;
- announcing benchmarks without caveats;
- treating agent-generated volume as progress;
- counting positive results while hiding failures;
- calling external interest “validation”;
- measuring models added without measuring models rejected.

## Institutional adoption signal

A funder, university, lab, or public-interest group adopting OpenAMP should be counted as meaningful only if they use or support at least one of:

- external benchmark audit;
- evidence certificate schema;
- data/model cards;
- review packet format;
- safe negative-result archive;
- pre-registration template;
- structured result intake;
- release governance.

## Final standard

OpenAMP adoption should mean serious people choose its process because they trust it more than alternatives.

Popularity without trust is not the goal.
