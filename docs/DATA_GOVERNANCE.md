# Data Governance

## Purpose

This document defines how OpenAMP Foundry should handle datasets, references, benchmarks, labels, preprocessing, licensing, and data-derived claims.

Data governance is scientific governance.

A strong model trained or evaluated on confused data is not strong. It is unreviewable.

## Prime rule

**No dataset may become authoritative until its source, license, labels, preprocessing, intended use, and known biases are documented.**

## Data classes

| Class | Meaning | Default release posture |
|---|---|---|
| Toy/demo data | Small artificial or safe example data for tests and demos. | Open. |
| Public redistributable data | Third-party data with clear redistribution permission. | Open with citation/license. |
| Public reference-only data | Publicly accessible but not redistributable. | Metadata/fetch instructions only. |
| Restricted data | Data with access limits, partner terms, or sensitive context. | Do not commit raw data. |
| Generated candidate data | Candidate sequences or panels generated by the project. | Review before release. |
| Qualified result summaries | External result summaries at safe abstraction level. | Review before release. |
| Negative-result data | Failed or inactive candidate outcomes. | Publish only when safe and licensed. |

## Data lifecycle

```text
source identification
  -> license review
  -> dataset card
  -> local fetch or import
  -> preprocessing record
  -> label audit
  -> leakage audit
  -> benchmark or reference use
  -> metric reporting
  -> periodic review
  -> deprecation if stale or unsafe
```

## Required dataset card

Every non-toy dataset should have a dataset card.

Minimum fields:

```yaml
dataset_id: stable-id
title: human-readable name
dataset_class: toy | redistributable | reference-only | restricted | generated | result-summary
source_name: name
source_url: url-or-private-reference
access_date: YYYY-MM-DD
license: license-or-terms-summary
redistribution_status: allowed | metadata-only | restricted | unknown
citation_required: true-or-false
label_definitions: description
intended_use: reference | benchmark | training | validation | demo | result-intake
not_for: list of prohibited uses
preprocessing_steps: list
known_biases: list
leakage_risks: list
safety_release_status: open | staged | restricted | do-not-release
reviewer: name-or-role
last_reviewed: YYYY-MM-DD
```

## Label governance

Labels must mean what the benchmark says they mean.

For AMP work, labels can be slippery:

- active under one organism does not mean broad-spectrum;
- non-hemolytic under one condition does not mean safe;
- inactive in one assay does not mean universally inactive;
- random decoy does not mean biological negative;
- known AMP does not mean clinically useful;
- literature-reported activity may lack comparable conditions.

Dataset cards must define labels precisely.

## Negative-set policy

Negative sets are a major benchmark risk.

Weak negatives can make the model look better than it is.

Every negative set should state:

- how negatives were chosen;
- whether they are random peptides, shuffled peptides, non-AMP proteins, inactive peptides, or synthetic controls;
- whether charge, length, composition, or similarity were matched;
- what shortcut the negative set might allow;
- what claim the benchmark cannot support.

## Leakage policy

Data leakage can occur through:

- exact sequence overlap;
- near-duplicate families;
- template variants;
- train/reference/test reuse;
- benchmark positives also used for novelty scoring;
- generated candidates derived from references;
- manually tuned thresholds after seeing benchmark results.

Leakage checks should be documented and automated where possible.

## Generated candidate data

Generated candidate lists are not automatically safe to publish.

Before releasing generated candidates, review:

- candidate purpose;
- novelty and similarity;
- safety-risk flags;
- release status;
- whether the list is screened or unscreened;
- whether release could enable misuse;
- whether metadata-only release is safer.

Generated candidates should not be committed as public artifacts merely because they are computational outputs.

## Result-summary data

Qualified external result summaries should preserve scientific meaning without turning the repo into an operational laboratory record.

A result summary may include:

- candidate ID;
- panel ID;
- endpoint category;
- result summary;
- quality/control interpretation supplied by the qualified partner;
- whether the result is interpretable;
- whether it supports recalibration consideration;
- limitations;
- release status.

Do not commit detailed operational protocols or sensitive partner data.

## Preprocessing policy

Preprocessing must be reproducible.

Record:

- input source;
- script or command;
- version;
- parameters;
- exclusions;
- deduplication method;
- normalization rules;
- output hash where practical.

Silent preprocessing changes are forbidden for benchmark data.

## Citation and license policy

Respect dataset creators.

Do not redistribute data merely because it is accessible online.

If redistribution is unclear:

- do not commit raw data;
- commit only metadata where safe;
- provide instructions for users to obtain data themselves if allowed;
- document access date and citation requirements.

## Benchmark data review

Before a dataset enters a gate benchmark, answer:

1. Is redistribution allowed?
2. Are labels well-defined?
3. Are negatives meaningful?
4. Are near-duplicates controlled?
5. Is charge/length/composition shortcut risk documented?
6. Does the benchmark answer the intended claim?
7. Is the benchmark card complete?
8. Is the result recorded in `docs/METRICS_CURRENT.md`?

## Data deprecation

Deprecate a dataset when:

- license status becomes unclear;
- leakage is discovered;
- labels are too ambiguous;
- better curated data exists;
- the dataset no longer matches its claimed use;
- safety review says release is inappropriate;
- a benchmark built on it becomes misleading.

Deprecation should be documented, not hidden.

## Data contribution checklist

A data contribution must include:

- dataset card;
- license or terms summary;
- citation requirements;
- intended use;
- preprocessing script or description;
- label definitions;
- known biases;
- leakage risks;
- safety-release status;
- tests or validation where applicable.

## Final standard

OpenAMP data should be boringly inspectable.

A serious outsider should know where every dataset came from, what it means, what it cannot prove, and whether it can legally and safely be reused.
